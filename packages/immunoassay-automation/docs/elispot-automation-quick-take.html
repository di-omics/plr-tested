Why drive it from a Pi through an agent harness

Every instrument here is PyLabRobot-compatible — open source. So instead of stitching together siloed vendor software, the whole line runs from a Raspberry Pi through one agent harness. That buys four things vendor UIs do not:

Scriptable

The protocol is code. It reads like code, diffs like code, and an SME can write and change it — no macro recorder, no per-vendor scripting dialect.

Closed-loop

Returning data feeds the next run. Background, saturation, and replicate scatter become concrete parameter deltas for the next plate, automatically.

Benchmarkable

The control software is small enough for a scientist to read, test, and benchmark — the same code qualifies the instrument that runs the assay.

No lock-in

One harness spans washer, liquid handler, and imager. Swap an instrument and the plan is unchanged; the site profile absorbs the difference.

What "automated" actually buys: gated, never-invented, transferable

Consistency is not just running the same steps — it is refusing to trust a run that should not be trusted. This is built as a QC-gated pipeline, not a macro:

  • Gate 0 — instrument readiness. Before a well is coated, the washer's dispense CV and aspiration residual are measured (a Rhodamine ladder). Not tight enough — the run stops before a sample is spent. The wash is the single biggest controllable source of ELISpot noise.
  • Gate 2 — plate validity, then response calls. The mitogen positive control must fire and the background must be low, or the plate is void. Only then is each antigen scored against background — by the conservative net-plus-fold rule, or the field's distribution-free resampling test (a permutation test on the replicate wells, Moodie 2010) — with replicate-CV trust and a saturation flag.
  • Never invent. Every reagent value carries its origin. The kit's antibody concentrations and the substrate development endpoint are marked unresolved and a real run is blocked until they are transcribed from the kit and set on the first plate.
  • Transferable & remote. The core is standard-library Python — a partner site runs a full simulation and reads a dossier before an instrument is unboxed. A plate that develops overnight resumes asynchronously once the imager has counted it.

Where the scientific judgment lives: the membrane

Most of ELISpot automates cleanly. The catch is the PVDF membrane the assay is read off. An engineer reading the protocol like code should stop at three steps and ask "does this survive a robot":

1. Probe clearance — a wash probe that rides too low scratches the membrane and prints a line of false spots; the safe height is taught per plate lot, not guessed.  2. No jet — a center-jet dispense lifts the capture-antibody layer; reagents go down the side wall at capped flow.  3. Wet-out discipline — the membrane stays wet from coat to develop, and the wash cannot be run harder "to be safe." These are guarded values in the code, not comments.

Status: honest about maturity

This is a working, simulation-first package — 39 offline tests, the full flow runs end to end and the gates demonstrably stop a bad run. No ELISpot instrument has been driven from it yet: the washer and imager integrations are the next build, and each hardware command is recorded as a plan, never as a claim that an instrument was driven. That distinction is the point.
